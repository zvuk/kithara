## Fix #1: downloader plan_impl() seek_pending deadlock

### Hypothesis
`HlsDownloader::plan_impl()` at downloader.rs:1174-1177 returns `PlanOutcome::Idle`
when `is_seek_pending()` is true. But `is_seek_pending` is only cleared after the
decoder completes the seek — which requires data from the downloader. Classic deadlock.

### Deep Investigation (COMPLETED)
Full trace of the seek path:

1. `Audio::seek()` → `timeline.initiate_seek()` → epoch=N, flushing=true, seek_pending=true
2. Worker: `step_seek_requested()` → `apply_seek_from_timeline()`:
   - `set_seek_epoch(N)` → clears segment_requests
   - `seek_time_anchor()` → `apply_seek_plan()` → clears segment_requests AGAIN
   - `timeline.complete_seek(N)` → **clears is_flushing** (but seek_pending stays true)
   - Transitions to `ApplyingSeek`
3. Worker: `step_applying_seek()`:
   - `source_is_ready_for_apply_seek()` → `source_ready_for_range()`:
     - `is_range_ready(range)` → range NOT ready
     - `phase_at(range)` → `SourcePhase::Waiting`
   - Enters `WaitingForSource` → returns `Blocked`
4. Worker stays in WaitingForSource, polling `phase_at()` which is pure check with NO side effects

Key findings:
- `phase_at()` (source.rs:805-821) does NOT submit demand requests
- `is_range_ready()` (source.rs:1091-1117) has a workaround that submits demand,
  but ONLY for ephemeral backends when committed_segment exists
- For mmap backends or after SeekLayout::Reset (segments cleared), demand may not be submitted
- Even when demand IS submitted, the main deadlock path is:
  - Demand submitted → downloader woken → `drain_demand_requests()` runs
  - BUT `poll_demand_impl()` checks `is_flushing()` (OK, false at this point)
  - Demand processed → segment fetched → committed → data available
  - Worker sees Ready → proceeds

### Actual Root Cause
The workaround in `is_range_ready()` is brittle and insufficient. The real problem is
architectural: the worker's FSM needs data to complete a seek, but the demand path
relies on a fragile side-effect in `is_range_ready()` with an `is_ephemeral()` guard.

### Decision: Remove workaround, implement proper architectural solution
The `is_range_ready()` workaround at lines 1106-1117 is tech debt.
Need proper demand submission from the seek FSM path.

### Solution: Replace `is_range_ready()` with `demand_range()` (IMPLEMENTED)

**Design decision**: `is_range_ready()` was redundant with `phase_at() == Ready` in all
implementations (HLS uses `range_ready_from_segments()` in both, File uses `contains_range()`
in both). The method was tech debt — a query with hidden demand side-effects.

**Changes (net zero methods on Source trait)**:
1. **Removed** `is_range_ready()` from Source trait, Stream<T>, SharedStream, HlsSource, FileSource, MemorySource
2. **Added** `demand_range()` — explicit non-blocking demand signal (replaces `is_range_ready`)
3. **Simplified** `source_ready_for_range()` — now just `phase_at(range) == Ready|Eof|Seeking`
4. **Added** `submit_demand_for_current_state()` in worker FSM — called from
   `step_waiting_for_source()` when phase shows data isn't ready
5. **HlsSource::demand_range()** — delegates to existing `queue_segment_request_for_offset()`
   (no `is_ephemeral()` guard, works for all backends)

**Test results after fix**:
- 87 passed, 20 failed (was 88/19) — but 4 "new" failures pass individually (contention noise)
- **Fixed tests**: stress_seek_lifecycle_with_zero_reset_mmap, stress_random_seek_read_hls_medium,
  concurrent_file::four_file_instances, concurrent_file::eight_file_instances (flaky)
- Remaining failures: other root causes (Groups 3, 4, 5-divergence)

### Status: DONE

## Fix #2: Sequential HLS session poisoning (5 tests, suite_light)

### Failing tests
1. `sequential_hls_warmup_does_not_poison_next_ephemeral_session` — HangDetector 1.2s
2. `sequential_hls_warmup_drop_only_does_not_poison_next_ephemeral_session` — HangDetector 1.2s
3. `sequential_hls_read_only_session_does_not_poison_next_ephemeral_session` — HangDetector 1.2s
4. `sequential_hls_stream_sessions_do_not_poison_next_ephemeral_session` — wait_range timeout 10s
5. `player_worker_hls_then_unavailable_mp3_then_mp3_recovery_ephemeral` — HangDetector 1.2s

### Pattern
- First session (disk store) succeeds
- Second session (ephemeral store) hangs
- Tests use completely separate servers, workers, temp dirs, stores
- Standalone ephemeral tests PASS (no prior session needed to trigger)

### Eliminated hypotheses
1. **Global buffer pools poisoning** — `GLOBAL_BYTE_POOL`/`GLOBAL_PCM_POOL` are OnceLock,
   initialized once and reused. No mutation after init. Not the cause.
2. **Shared reqwest::Client** — Each session creates its own `HttpClient::new()` →
   fresh `reqwest::Client::builder().build()`. Not shared globally. Not the cause.
3. **MemAssetStore::resource_state() always returns Missing** — True for raw MemAssetStore,
   but `CachedAssets` layer caches committed state via `cached_state()`. Confirmed by
   passing unit tests like `hls_phase_ready_when_range_ready`. Not the primary cause.
4. **Worker thread not joining** — `Backend::drop` joins the backend thread.
   `AudioWorkerHandle::shutdown()` doesn't join, but worker threads are independent.
5. **reqwest Client bound to wrong runtime** — Each session creates Client in its
   context, connections are per-request. Not the cause.

### Active hypothesis: CurrentThread runtime deadlock via `recv_outcome_blocking`

**Key finding**: `warm_hls_worker()` does NOT call `audio.preload()` before the first
`audio.read()`. Audio starts with `preloaded=false` (audio.rs:868).

When `preloaded=false`:
- `audio.read()` → `recv_outcome()` → `recv_outcome_blocking()` (audio.rs:471)
- `recv_outcome_blocking` loops: `try_pop() || park_timeout(100μs)` — blocks OS thread

The test uses `#[kithara::test(tokio, ...)]` without `multi_thread` → **CurrentThread** runtime.
The HTTP server is spawned via `tokio::spawn` on this runtime (http_server.rs:56).

When `recv_outcome_blocking` parks the main thread:
- The tokio runtime's only thread is blocked
- Server tasks can't run → can't accept/respond to HTTP requests
- Backend thread sends requests → server doesn't respond → no data
- Worker thread waits for data → no chunks produced
- `recv_outcome_blocking` never gets data → deadlock

**Why first session works**: Race condition. Between `Audio::new().await` and the first
`audio.read()`, the backend/worker pipeline may complete one fetch cycle during the
`.await` points inside `Audio::new()`. This is timing-dependent and works for the first
session (low thread contention) but fails for the second (more threads, cleanup overhead).

**Comparison with PASSING tests**:
- `read_some()` calls `resource.preload()` → `audio.preload()` → `preloaded=true`
- With `preloaded=true`, `recv_outcome()` uses `try_pop()` (non-blocking), returns `Empty`
- `audio.read()` returns 0 immediately → `sleep(10ms).await` yields to runtime → server runs
- `player_worker_hls_then_mp3_reopen_keeps_backward_seek_ephemeral` PASSES because it
  uses `read_some()` (with preload), not `warm_hls_worker()` (without preload)

**Test 4 (sequential_hls_stream_sessions_do_not_poison_next_ephemeral_session)**:
Uses raw `stream.read()` (sync, blocks thread). Same CurrentThread deadlock, different path.
Error: `wait_range budget exceeded: range=0..4096 elapsed=10s timeout=10s`.

### Fix applied
1. Added `audio.preload()` before each `audio.read()` in `warm_hls_worker` and
   `warm_hls_worker_without_seek` — makes recv non-blocking, allows tokio runtime to
   process server tasks between iterations.
2. Added `multi_thread` to `sequential_hls_stream_sessions_do_not_poison_next_ephemeral_session`
   because raw `stream.read()` inherently blocks the thread (no preload equivalent for Stream).

**Verification**: multi_thread confirmed fix (passed in 0.26s). preload() confirmed fix
for all 4 Audio-based tests (passed in 1.15s). All 5 tests pass.

**Regression check**: suite_light: 474 passed, 4 failed, 3 ignored.
Before fix: 468 passed, 9 failed. The 4 remaining are pre-existing (HLS seek contract, Task #2).

### Status: DONE

## Fix #3: OnceCell deadlock in handle_batch (4 tests, suite_light)

### Failing tests
1. `seek_to_segment_boundary_reads_correct_segment_case_3` (pos=400000)
2. `hls_stream_seek_to_segment_start_case_3` (pos=400000)
3. `seek_burst_then_tail_read_stays_contiguous_disk`
4. `seek_burst_then_tail_read_stays_contiguous_ephemeral`

### Eliminated hypotheses
1. **CurrentThread runtime deadlock** — Added `multi_thread`, still fails. Reverted.
2. **Tracing/compilation timing mask** — Test passes with RUST_LOG or code changes. Masks race, not a fix.
3. **Batch not including segment 2** — Debug shows `batch_end=3`, all 3 planned.
4. **Commit rejection** — Debug shows commits for seg 0, 1 succeed. Seg 2 commit never reached.
5. **Downloader stuck in handle_idle** — Debug shows demand is taken at `[DL-DEMAND]`, so it's not idle.

### Root cause (CONFIRMED via eprintln instrumentation)

`handle_batch` → inner commit loop → `try_poll_demand` → `fetch_and_commit_demand` → deadlock:

1. Batch launches parallel fetches for segments 0, 1, 2 via `FuturesUnordered`
2. After committing segment 1, `try_poll_demand` picks up demand for segment 2
3. `fetch_and_commit_demand` → `io.fetch(plan)` → `load_media_segment_with_source_for_epoch`
   → `cell.get_or_try_init(...)` where `cell` is shared `Arc<OnceCell<SegmentLoad>>`
4. Batch's segment 2 fetch already called `cell.get_or_try_init(...)` first — it holds the init lock
5. Demand's `get_or_try_init` WAITS for the batch's init to complete
6. But batch's segment 2 future is in `FuturesUnordered` which is NOT being polled (outer loop blocked)
7. **Classic single-thread executor deadlock via OnceCell coalescing**

Debug trace:
```
[DL-DEMAND] got req: seg=0 variant=1 epoch=0   ← initial demand from Hls::create
[DL-COMMIT] seg=0 variant=1 epoch=0
[DL-BATCH] cursor=1 batch_end=3 variant=1 num_segments=3
[DL-COMMIT] seg=1 variant=1 epoch=0
[DL-DEMAND] got req: seg=2 variant=1 epoch=0   ← demand taken, then HANGS forever
```

### Fix: Remove try_poll_demand from handle_batch inner loop

The `try_poll_demand` inside `handle_batch`'s commit loop can deadlock with pending batch
futures when both target the same segment (via OnceCell coalescing). Demand processing is
still handled by:
- `drain_demand_requests()` at the top of each main loop iteration
- `try_poll_demand` in `wait_while_throttled()`

Impact: demand latency increases by batch duration (~16ms for local server, ~100ms remote).
This is acceptable — demand is a best-effort optimization, not a correctness requirement.

### Verification
- All 4 previously failing tests: PASS
- suite_light: 478 passed, 0 failed, 3 ignored (was 474/4/3)

### Status: DONE

## Fix #4: Encrypted HLS stress test HangDetector timeout (1 test, suite_heavy)

### Failing test
`stress_random_seek_read_hls_encrypted_medium` — HangDetector 1s

### Eliminated hypotheses
1. **OnceCell deadlock (same as Fix #3)** — Each segment has its own media OnceCell; no inter-future
   deadlock found. Key fetch uses `fetch_atomic_internal` (no OnceCell).
2. **CurrentThread runtime deadlock** — Downloader has its own CurrentThread runtime; test server
   runs on the test's runtime. Network requests go through TCP (independent runtimes).
3. **reqwest connection pool exhaustion** — 50 concurrent HEAD requests should be handled by
   connection pooling.

### Root cause (CONFIRMED via eprintln instrumentation)

`calculate_size_map` sends HEAD requests for ALL 50 segments in parallel before the downloader
can process any data. For encrypted segments, the test server had NO explicit HEAD handler, so
axum fell through to the GET handler which:
1. Generated 100KB of segment data
2. Encrypted it with AES-128-CBC
3. Returned the full response (HEAD strips body, but work is done)

50 segments × 100KB encryption = 5MB of AES encryption on a single-threaded test server.
This took ~1.29s, exceeding the 1s HangDetector timeout.

Timeline:
- T=0: Downloader starts, HangDetector deadline = T+1s
- T+1.29s: `calculate_size_map` completes (50 HEAD requests processed)
- T+1.34s: First batch (seg 0) committed, `hang_reset!()` called
- T+1.34s: Second batch plans built (seg 1-3)
- T+1.34s: `hang_tick!()` fires — deadline (T+1s) already passed by 0.34s
- Non-encrypted medium (same segment count) PASSES because `generate_segment` is fast (no crypto)

### Fix: Add efficient HEAD handler to test server

Added `serve_segment_head_default()` that computes Content-Length without generating or
encrypting segment data. For encrypted segments, returns `plaintext_size + (16 - plaintext_size % 16)`
(PKCS7 padding formula). Registered HEAD handler for ALL cases, not just `head_reported_segment_size`.

### Verification
- `stress_random_seek_read_hls_encrypted_medium`: PASS (3/3 runs, 2.1-2.2s each)
- All 7 `stress_random_seek_read_hls` variants: PASS (16.8s)
- suite_light: 478 passed, 0 failed, 3 ignored (no regression)

### Status: DONE

## Fix #5: Multi-instance File divergence (4 tests, suite_heavy)

### Failing tests
1. `eight_file_instances` — instance 6 reads 32,350 samples instead of 16,501,342
2. `mixed_two_file_two_hls` — timeout 10s
3. `mixed_four_file_four_hls` — timeout 10s
4. `four_file_instances` — flaky (occasionally fails)

### Root cause
`AudioTestServer` did not support HTTP Range requests. When the file downloader switches
from sequential download to gap-filling (demand triggered by decoder reading before
downloader has written data), it sends Range requests. Server returned 200 (full file)
instead of 206 (partial).

`Writer::with_offset(stream, res, cancel, range.start)` writes the body starting at
`range.start` offset in the resource. But the body is the FULL file from byte 0. So
file byte 0 gets written at resource offset `range.start` — data corruption.

With parallel gap-fill (`FuturesUnordered` batch of 4 ranges), one fetch completes and
commits resource (no gaps left because full file data overwrote everything). Other
in-flight fetches then try to write to committed resource → `res.fail(...)` →
resource marked FAILED. Decoder hits failed resource → premature EOF.

### Fix
Added HTTP Range support to `AudioTestServer::serve_bytes()`:
- Parses `Range: bytes=start-end` header
- Returns 206 Partial Content with correct `Content-Range` header
- Falls through to 200 for requests without Range header

### Verification
- `eight_file_instances`: 3/3 runs PASS (all 8 instances = 16,501,342 samples)
- All 11 multi_instance tests: PASS (56s)
- suite_light: 478 passed, 0 failed (no regression)

### Status: DONE

## Fix #6: DRM test failures (11 tests, suite_heavy)

### Failing tests (DRM-only, non-DRM equivalents pass)
1. `live_ephemeral_revisit_sequence_regression_drm`
2. `live_ephemeral_small_cache_playback_drm`
3. `live_ephemeral_small_cache_seek_stress_drm`
4. `live_real_stream_fixed_seek_window_regression_drm`
5. `live_real_stream_random_seek_prefix_regression_drm`
6. `live_stress_real_stream_seek_read_cache_drm_ephemeral`
7. `live_stress_real_stream_seek_read_cache_drm_mmap`
8. `stress_seek_during_abr_switch_real_decoder_drm`

### Error patterns observed
1. **"unexpected end of file"** — from Symphonia IsoMp4Reader::seek. Seeks near end of stream.
2. **"unexpected end of bitstream"** — from AAC decoder. Corrupt/misaligned data after seek.
3. **"seek past EOF"** — from Stream::seek(). `total_bytes` dramatically wrong (e.g., 3978614 for a 25M stream).

### What's unique about DRM
- `calculate_size_map()` uses HEAD → returns ENCRYPTED sizes (larger by PKCS7 padding: 1-16 bytes/segment)
- After download + decryption, actual sizes are smaller
- `reconcile_total_bytes()` adjusts `total_bytes` incrementally per segment
- With 37 segments, total padding drift = 37-592 bytes

### Hypotheses tested

#### H1: ReadSeekAdapter dual-handle bug (CONFIRMED as real bug, PARTIAL fix)
**Problem**: In `new_direct()`, when `config.byte_len_handle` is provided, the adapter creates
its OWN `byte_len: Arc<AtomicU64>` (probed from source). `SymphoniaInner.byte_len_handle` stores
the config's handle. `update_byte_len()` writes to config handle, but Symphonia reads from
adapter's handle (never updated).

**Fix**: Added `ReadSeekAdapter::new_seek_disabled_shared()` that accepts an external byte_len
handle. In `new_direct()` and `probe_with_seek()`, when config provides a handle, the adapter
uses the SAME Arc. Now `update_byte_len()` and `MediaSource::byte_len()` are in sync.

**Result**: 1 additional test passing (6 pass vs 5 before). But remaining tests still flaky —
different tests pass/fail each run, suggesting timing-dependent issues.

#### H2: update_decoder_len_for_seek() too restrictive (TESTED, REVERTED)
**Problem**: `update_decoder_len_for_seek()` only updates byte_len when `base_offset == 0`.
After ABR switch, decoders with `base_offset > 0` never get byte_len updates.

**Fix attempted**: Removed `base_offset == 0` guard, computed `len.saturating_sub(base_offset)`.

**Result**: REGRESSION — 12 failures (was 5). The shared factory handle is used by ALL decoder
instances. Storing `total_bytes - base_offset` into the shared handle after ABR switch
corrupts the value for other contexts. REVERTED to original guard.

#### H3: cascade_contiguity for segment offset drift (IMPLEMENTED, no-op)
**Problem**: After DRM reconciliation, later segments' byte_offsets in DownloadState become
stale (based on encrypted sizes of earlier segments).

**Fix**: Added `cascade_contiguity()` to DownloadState — walks forward from a committed
segment and relocates subsequent segments to maintain contiguity. Called after `push()` in
`commit_segment()` and after `relocate_segment()` in `segment_loaded_for_demand()`.

**Result**: No cascades triggered in practice — sequential batch commits already produce
contiguous layouts. The fix is defensive but doesn't address the actual failures.

#### H4: Symphonia moof byte positions become invalid (INVESTIGATED, not the cause)
**Investigation**: Symphonia's IsoMp4Reader caches absolute byte positions of moof atoms
during sequential parsing (stored in `MoofAtom::moof_base_pos`). No invalidation mechanism.

**Finding**: NOT the cause. During sequential reading, moof positions are recorded at ACTUAL
read positions (based on decrypted segment sizes in DownloadState). The positions are correct
from the start. The virtual stream layout change from reconciliation only affects total_bytes,
not individual segment positions — segments in DownloadState already use actual sizes.

#### H5: total_bytes set to raw variant total after ABR switch (LIKELY ROOT CAUSE, not yet fixed)
**Problem**: After ABR variant switch to segment 0 (non-midstream), `refresh_variant_total_bytes()`
sets `total_bytes = new_variant_total` (from encrypted HEAD sizes). If the new variant is
lower bitrate (~4M vs ~25M), total_bytes drops dramatically. The stream position from the
old variant (22M) is now past the new total (4M) → "seek past EOF".

`calculate_effective_total()` only adjusts for `is_midstream_switch`. For seek-driven switches
where `segment_index == 0`, `is_midstream_switch = false`, so raw total is used.

**For DRM specifically**: the encrypted total is even MORE wrong (overestimate by PKCS7 padding).
But the dramatic total_bytes drop is NOT DRM-specific — it should also affect non-DRM ABR
switches. Needs further investigation: why non-DRM ABR tests pass.

### Tests always passing (simpler scenarios, no seeks or ABR)
- `live_real_drm_playback_smoke` — sequential playback, no seeks
- `seek_sequence_from_log_real_stream_drm` — pre-recorded seek sequence
- `live_real_stream_seek_resume_native_drm` — single seek

#### H6: init_len not preserved on re-download → reconciliation drift (CONFIRMED, FIXED)
**Problem**: When LRU evicts a segment and demand re-download fetches only the media
(no init), `commit_segment` sets `init_len = 0`. If the original commit had `init_len > 0`
(variant switch or segment 0), `reconcile_segment_size` gets `actual_size = media_len`
instead of `actual_size = init_len + media_len` → `total_bytes` shrinks → premature EOF.

**Fix part 1 — init_len preservation**: In `commit_segment`, when `fresh_init_len == 0`,
look up existing DownloadState entry's `init_len` and preserve it. Also preserve `init_url`
from existing entry when init_len is inherited.

#### H7: Demand re-download without init → infinite retry loop (CONFIRMED, FIXED)
**Problem**: H6 fix preserves `init_len > 0` from existing metadata. But `segment_resources_available`
checks init resource via `backend.resource_state()`. For ephemeral backends, `MemAssetStore::resource_state()`
returns `Missing`; `CachedAssets` checks LRU, but init resource was evicted and NOT re-fetched.
So `segment_resources_available` always returns false → `segment_loaded_for_demand` returns false →
demand re-download loops infinitely → HangDetector fires.

**Fix part 2 — demand_init_evicted**: Added `demand_init_evicted()` method that checks
if a previously-committed segment's init resource has been evicted from ephemeral cache.
Used in `build_demand_plan` to set `need_init = true`, ensuring init is re-fetched alongside
media on demand re-downloads. This breaks the infinite retry loop.

**Both fixes together**: init_len preserved for correct reconciliation + init resource
re-fetched when evicted to prevent livelock.

set to raw variant total after ABR switch) or Symphonia's seek algorithm with fMP4.

Failing on main AND with our changes (same 7):

- live_ephemeral_small_cache_seek_stress_drm

Passing:

### Status: H6+H7 FIXED. DRM seek is pre-existing bug on main.
