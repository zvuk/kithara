## Root Cause Analysis

### ROOT CAUSE 1: DRM `total_bytes` drift (tests #1-4)
**The fundamental problem**: `calculate_size_map()` sends HEAD requests → gets ENCRYPTED segment sizes.
After download + AES-128-CBC decryption, actual sizes are smaller by PKCS7 padding (1-16 bytes/segment).
`reconcile_total_bytes()` adjusts incrementally per committed segment, but `total_bytes` starts from
the encrypted total. With 37 segments × 1-16 bytes padding = 37-592 bytes overshoot.

When seeking near end of stream (e.g., 205.81s of 220s), the decoder's seek target resolves to a
byte position near `total_bytes`. But actual data ends `drift` bytes earlier. Symphonia's IsoMp4Reader
tries to read past actual data → "unexpected end of file".

**Why non-DRM is fine**: Without DRM, HEAD sizes = actual sizes. No drift. total_bytes is exact.

### ROOT CAUSE 2: DRM ABR hang (test #7)
After ABR switch during seek, the decoder enters `recv_outcome_blocking` waiting for data.
But the seek/ABR interaction causes a state where the worker thread can't make progress:
- Exact seek fails (due to ROOT CAUSE 1)
- Error propagation path fails to unblock the consumer
- HangDetector fires after 1s

### ROOT CAUSE 3: Unit test expectation stale (test #9)
Commit 7bb1bc7a8 changed `update_decoder_len_for_seek()` to always update byte_len
(removed `base_offset == 0` guard). But the test `#[case::after_abr_switch]` still expects
0 byte_len updates. The test expectation should be changed from 0 to 1.

### ROOT CAUSE 4: `prefer_anchor_skip` regression (tests #5-6)
The autoresearch `prefer_anchor_skip` flag is sticky — once a seek fails and triggers
decoder recreation + software skip, ALL subsequent seeks bypass exact decoder seek.
For non-DRM streams, exact seek works fine, so the recreation path introduces
unnecessary decoder churn and eventually fails at high seek positions.

## Fix Plan

### Fix 1 (trivial): Update unit test expectation
Change `expected_byte_len_updates` from 0 to 1 in `#[case::after_abr_switch]`.

### Fix 2 (core): Fix DRM `total_bytes` drift
**Approach**: After DRM reconciliation of ALL segments, recalculate `total_bytes` as
sum of actual (decrypted) sizes instead of relying on incremental adjustments.
OR: compute padding at HEAD time using PKCS7 formula: `encrypted_size - plaintext_size ≤ 16`.

### Fix 3 (remove autoresearch): Revert `prefer_anchor_skip`
The autoresearch changes are a workaround that trades correctness for non-DRM paths.
Once ROOT CAUSE 1 is fixed, DRM seeks near EOF should work without special skip logic.

### Fix 4 (DRM ABR hang): Proper error recovery in seek path
When exact seek fails, the error should propagate cleanly to unblock `recv_outcome_blocking`.
This may be a downstream effect of ROOT CAUSE 1 — fixing total_bytes drift should also fix this.

## Codex Analysis (Symphonia internals)

### How Symphonia fMP4 seek works:
1. `IsoMp4Reader::seek_track_by_ts` does LINEAR scan of known moof segments
2. If target timestamp not found in known moofs → calls `try_read_more_segments()`
3. `try_read_more_segments()` reads forward trying to parse more moof/mdat pairs
4. If it reaches EOF before finding target timestamp → "unexpected end of file"

### Key insight: TIMESTAMP-DOMAIN MISMATCH
The failure is NOT a byte-layout problem. `total_bytes` and `max_end_offset` are correctly reconciled.
The problem is: **Symphonia scans ALL moofs forward looking for timestamp 205.81s, but the fMP4
sample timeline doesn't extend to that timestamp** → Symphonia reaches EOF.

Possible causes:
1. **HLS playlist time vs fMP4 sample time drift**: `#EXTINF` durations are nominal,
   actual sample durations in moof/trun atoms may be slightly different. Over 37 segments,
   cumulative drift can cause HLS time 205.81s to be past the last MP4 sample.
2. **After ABR switch**: decoder is recreated with a new format reader. The new reader's
   time base starts from the segment where recreation happened. If seek target is expressed
   in absolute playlist time but reader expects segment-relative time, mismatch occurs.
3. **Multi-track**: fMP4 might have tracks that don't cover the full timeline
   (unlikely for audio-only HLS).

### Why DRM but not non-DRM?
Both DRM and non-DRM use the same fMP4 structure. BUT:
- DRM tests use the `slossless` variant (FLAC in fMP4) which has different frame sizes
- Non-DRM tests might use AAC variants with different timing characteristics
- The specific random seed produces seek targets that happen to fall in the drift zone
  for FLAC but not for AAC
- OR: DRM reconciliation timing changes when segments become available, affecting
  which moofs Symphonia has parsed before seek

### Next steps
1. **Verify hypothesis**: Add tracing to capture Symphonia's known moof timeline range
   vs the seek target time
2. **Fix approach A**: Clamp seek targets to Symphonia's known timeline
   (e.g., `min(target, decoder.duration() - epsilon)`)
3. **Fix approach B**: After exact seek failure, fallback to seek to last known timestamp
   + software skip (but per-attempt, NOT sticky)
4. **Fix approach C**: Cap seek targets to `duration - segment_duration` to avoid
   edge-of-stream seeks

For streams after midstream ABR switch:
- Decoder starts at base_offset (middle of virtual stream)
- ReadSeekAdapter wraps Stream starting from base_offset
- Symphonia sees init.mp4 + segments 3-36 as a contiguous stream

- Seek to 205.81s should find segment 34's moof

1. The moof scan uses `is_seekable()` to decide whether to seek backward — since we

2. The scan compares timestamps differently than expected (e.g., segment start vs end)
3. Off-by-one in the fragment loop boundary detection

Both use the same ABR switch path, but DRM tests consistently fail. Possible cause:
- DRM encryption delays → slightly different ABR switch timing → different base_offset
- Different number of segments in the prefix → different byte_len/position relationships
- OR: non-DRM tests happen to not trigger the edge case (different random seed behavior)

## FINAL ROOT CAUSE SUMMARY

### 3 distinct bugs:

#### Bug 1 (pre-existing): Symphonia fMP4 seek fails near EOF after midstream ABR switch
- **Tests**: #1-4, #7 (all DRM)
- **Root cause**: Symphonia's `IsoMp4Reader` can't seek to certain timestamps in fMP4 streams
  after decoder recreation at a non-zero base_offset. The linear moof scan reaches EOF.
- **Why DRM-specific**: Timing/offset differences between DRM and non-DRM ABR switches cause
  the failing seek targets to only appear in DRM tests (random seed + ABR timing dependent).
- **Fix approach**: Add a **per-attempt** seek fallback (not sticky). When exact seek fails
  with "unexpected end of file", fall back to seeking to a capped position or the nearest
  known segment boundary. This is essentially what autoresearch's `recreate_init_decoder_for_anchor_skip`
  does, but done correctly (per-attempt, not sticky, not full decoder recreation).

#### Bug 2 (autoresearch regression): Sticky `prefer_anchor_skip` breaks non-DRM seeks
- **Tests**: #5-6 (HLS non-DRM)
- **Root cause**: `prefer_anchor_skip` is set once and never cleared. Causes ALL subsequent
  seeks to bypass exact decoder seek, even when it would succeed.
- **Fix**: Do NOT apply autoresearch changes. They are not on main and should be discarded.
  If a fallback is needed, make it per-attempt only.

#### Bug 3 (pre-existing): Stale unit test expectation
- **Tests**: #9
- **Root cause**: Commit 7bb1bc7a8 changed `update_decoder_len_for_seek()` to always call
  `update_byte_len` regardless of base_offset. But the test case `after_abr_switch` still
  expects 0 byte_len_update calls. Should expect 1.
- **Fix**: Change `expected_byte_len_updates` from 0 to 1 in the test.
