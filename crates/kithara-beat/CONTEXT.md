# kithara-beat — Context

Detailed contracts and invariants for the kithara-beat crate; the README is the overview.

## Contract

- Input: **whole-track mono f32 PCM at 22 050 Hz**. Decoding, downmix, and
  resampling are the caller's responsibility — this crate has no decoder or
  resampler dependencies and does no I/O.
- Output: `RawBeats { beats, downbeats }` — positions in **seconds**, sorted,
  deduplicated, downbeats snapped to the nearest beat. Grid cleanup /
  source-frame conversion is the consumer's job (`kithara-audio`).
- Models load from bytes (`BeatThis::from_model_bytes`) — the caller decides
  embed vs file vs download.

```rust
let mut bt = BeatThis::from_model_bytes(mel_bytes, beat_bytes)?;
let raw: RawBeats = bt.analyze(&mono_22050)?;
```

## Pipeline

1. `mel.rs` — mel spectrogram via `mel_spectrogram.onnx` (rten with `fft`):
   input `"audio_pcm"` `[1, N]` → `[1, T, 128]`; 128 bands (30 Hz–10 kHz),
   window 1024, hop 441 → 50 fps, log scaling `ln(1 + 1000·x)`. The ONNX mel
   (not hand-rolled DSP) is what guarantees parity with the training pipeline.
2. `inference.rs` — chunked inference: windows of 1500 frames (30 s), 6 border
   frames trimmed per side, overlaps aggregated keep-first.
3. `postprocess.rs` — minimal peak picking (no DBN): a frame is a beat if it
   is the max of a 7-frame window (±3 = ±60 ms) and its logit > 0; equal
   plateaus collapse to their centre; each downbeat snaps to the nearest beat.

The pipeline constants (chunk 1500, border 6, stride 1488, 50 fps, hop 441,
threshold 0, ±3 window) are **frozen**: they define numerical parity with the
Python reference and must be folded into any analysis cache key by consumers.
Changing any of them invalidates the golden fixtures.

## Validation

`cargo test -p kithara-beat --features embed-small-model` runs the golden
parity test: F-measure ≥ 0.99 at the ±70 ms MIR window for beats and
downbeats against fixtures generated by Python `beat_this` v1.1.0 (see
`tests/fixtures/README.md` for provenance and why the small model is not held
to F = 1.0). The parity test takes a couple of minutes (whole-track NN
inference; the dev profile builds rten with `opt-level = 3`).
